# WikiSTEM

#### Description:

WikiSTEM is a Flask web application that gives high-school STEM students in northern Peru a public, permanent home for the research and projects they would otherwise submit once and never see again. The interface is entirely in Spanish, because the audience is real: International Baccalaureate and secondary students from schools in Piura. A student can publish an Extended Essay, an Arduino build, or a science-fair prototype; receive automated feedback on it; and have it appear in a searchable, school-attributed catalogue alongside the work of their peers. The tagline sums up the goal — *"Donde la ciencia estudiantil se hace pública."*

The platform is organised around two **tracks**. The **Research** track is for academic writing (IB Monografías, Internal Assessments, original papers) and centres on an uploaded PDF, an abstract, and metadata such as IB subject and word count. The **Projects** track is for hands-on builds (software, robotics, engineering prototypes) and centres on images, an optional embedded main PDF, supplementary documents, and a code snippet. Every submission moves through the same lifecycle: a student submits it (status `pending`), an automated content-moderation pass runs, and an administrator finally approves or rejects it. Only `approved` work is publicly visible.

A distinctive feature is the **two-stage Gemini integration**. When a submission is created, `run_ai_moderation` performs a fast safety check that can auto-flag inappropriate content. When an admin later approves it, `run_ai_review` produces a constructive, track-specific quality report — scoring argumentation, methodology, originality, and so on — rendered in an "AI panel" on the detail page. Both calls are deliberately fault-tolerant: they run under hard timeouts and never raise into the request, so a slow or failed Gemini call can never block a student's submission or an admin's decision.

Visitors get a personalised homepage feed, a paginated **browse** page with category filters and sort options, and a school **leaderboard** built from a single conditional-aggregation SQL query. **Search** is layered: a SQLite FTS5 full-text index handles exact and prefix matches, a category `LIKE` pass catches subject names, and a `rapidfuzz` fallback tolerates misspellings ("karna" → "karina"). Logged-in users get like buttons (AJAX), a private dashboard showing the moderation status of their own work, and a public profile page.

Account creation uses a **verify-then-create email flow**. Rather than writing an unverified `User` row and cleaning it up later, registration validates the form, hashes the password, packs the entire pending registration into a Fernet-encrypted, 24-hour token, and emails that token as an activation link — at this point no database row exists. The account is created only when the recipient opens the link (`/verify-email/<token>`), which is the single place a student `User` is ever inserted. Because Fernet gives confidentiality, integrity, and a built-in time-to-live in one primitive — and its output is URL-safe base64 that rides cleanly in a URL path — the design needs no `email_verified` column, no schema change, and no background job to sweep expired sign-ups: a stale or tampered link simply fails to decrypt and is rejected. Mail is delivered over SMTP through a dedicated Gmail mailbox using the standard-library `smtplib`, configured entirely through `MAIL_*` environment variables (the password is a Gmail App Password, never the account password). This proves a student controls the address they register with, while keeping unconfirmed registrations from ever touching the database.

**What each file does:**

- **`app.py`** is the application core — the Flask app factory and every route. It wires up the extensions (SQLAlchemy, Flask-Login, CSRFProtect, Flask-Limiter, Talisman, ProxyFix), defines the auth flow including the email-verification route that creates the account, the submit/browse/detail/search routes, the like and delete endpoints, the admin moderation queue, and the admin two-factor-auth flow.
- **`models.py`** defines the SQLAlchemy schema: `User`, `Submission`, `SubmissionLike`, `ViewHistory`, and `ModerationLog`. It also registers a `PRAGMA foreign_keys=ON` listener so SQLite actually enforces the cascade deletes the relationships declare.
- **`helpers.py`** holds the security-sensitive utilities: Fernet encryption/decryption, the email hashing used for login lookups, the verify-then-create registration-token helpers and their `smtplib` email delivery, Markdown rendering through a strict Bleach allowlist, slug generation, tag sanitising, the FTS query sanitiser, the fuzzy-search function, and the hardened image/document upload validators.
- **`forms.py`** contains the eight WTForms classes that define and validate every form, plus the category, school, and grade choice constants. Validators enforce things like the 128-character password cap and required academic-integrity confirmation.
- **`ai_review.py`** isolates all Gemini interaction: the prompts, the JSON contracts, and the `run_ai_review` / `run_ai_moderation` runners with their timeouts and error handling.
- **`recommendations.py`** scores submissions against a user's view and like history (favoured categories, preferred track, liked tags, recency, popularity) to build the personalised two-column homepage feed, with a sensible fallback for new and guest users.
- **`config.py`** provides environment-aware configuration objects (development vs production), reading secrets from the environment.
- **`init_db.py`** is the one-time setup script: it creates the tables, the FTS5 virtual table and its sync triggers, the upload directories, and seeds the admin account.
- **`pixel_glyphs.py`** generates the coordinates for the pixel-art "WikiSTEM" wordmark used in the branding, exposed to templates as Jinja globals.
- **`templates/`** holds the Jinja2 pages and reusable partials/macros; **`static/`** holds the CSS and the JavaScript (`main.js`) that powers the like buttons, live counters, image previews, and mobile navigation.

**Design choices.** The decision that shaped almost everything was to treat **security as non-negotiable** rather than as a feature. Passwords are hashed with PBKDF2 (explicitly pinned, because Werkzeug 3 silently changed its default to scrypt), and login runs a constant-time check even for unknown emails so response timing can't be used to enumerate accounts. Personally identifiable data — email and bio — is Fernet-encrypted at rest; because an encrypted column can't be queried, I store a separate SHA-256 `email_hash` purely for login lookups. Uploads are validated by magic bytes rather than by file extension, images are run through `exif_transpose` and capped to defeat decompression-bomb attacks, and user documents are stored **outside** the static folder and served only through a route that verifies ownership and forces `nosniff` — so a malicious `.md` can't be sniffed into executable HTML. CSRF is enforced globally, and the login `next` parameter is validated against a battery of open-redirect tricks. I also added admin TOTP two-factor authentication beyond the original spec.

Two other choices were deliberate trade-offs. First, I chose **SQLite with FTS5** instead of Postgres + a search engine: the corpus is a single school community, so a file-based database with a built-in full-text index keeps the project genuinely free to host while still delivering real search. Second, I store category and type values **in Spanish in the database** rather than as English keys mapped at render time. This keeps the data human-readable and the templates trivial; the cost is that internal state keys (`status`, `track`) stay in English and are mapped to Spanish labels only at the moment of display, which I considered the cleaner boundary. Finally, AI moderation runs at submission but **never blocks** it — a student should never be punished by a third-party API outage — so failures degrade gracefully into the admin's manual review queue.
